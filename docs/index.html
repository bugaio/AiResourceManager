为什么分层？

03

部署的两大魔法

软链接 vs 深度合并 — 两种截然不同的资源投递方式

魔法一：软链接（Symlink）

Skill 和 Agent 部署时，系统在目标文件夹创建一个 符号链接。

📎

源文件位置

~/.aiManager/skills/{UUID}/

🎯

目标位置

~/project/.claude/skills/my-skill →

💡
为什么用软链接而不是复制？

软链接是"活的"。你改了源文件，目标位置立刻看到变化。不需要重新部署。删除软链接只删链接，不删源文件。这就像在桌面放一个快捷方式，删快捷方式不会删程序本身。

魔法二：深度合并（DeepMerge）

Config 类型要把一个 JSON 片段合并进一个已有的完整配置文件。这是 AiResourceManager 最复杂的逻辑。

CODE 

// 资源片段 (要部署的)
{ "mcpServers": {
    "my-tool": { "command": "node" }
}}

// 目标文件 (已有的)
{ "mcpServers": {
    "other": { "command": "py" }
}, "theme": "dark" }

// 合并结果
{ "mcpServers": {
    "other": ..., "my-tool": ...
}, "theme": "dark" }
PLAIN ENGLISH

这是要部署的新配置片段

想添加一个叫 my-tool 的 MCP 服务器

这是目标配置文件原本的内容

已经有一个 other 服务器和 theme 设置

合并后 — 两个 mcpServers 都被保留

my-tool 被加进 mcpServers 对象里

theme 完全不受影响，保持不变

Prompt 部署：追加策略

Prompt 资源使用第三种策略——追加。每个资源用 UUID 做唯一分隔符，嵌入目标 .md 文件末尾：

skill → 部署方式：软链接到目标目录
agent → 部署方式：软链接 .md 到目标目录
config → 部署方式：深度合并到 JSON/YAML/TOML
prompt → 部署方式：追加块（带 UUID 分隔符）到 .md

关键代码在 deploy.go 第 1516-1518 行：

CODE 

func isMergeResourceType(t string) bool {
    return t == "config" || t == "prompt"
}
PLAIN ENGLISH

定义一个判断函数：资源类型是否需要合并部署？

Config 和 Prompt → 合并写入文件（是）

Skill 和 Agent → 软链接部署（否）

🔑
一行代码决定行为

整个部署系统的"类型判断真相"就藏在这一行。isMergeResourceType 是整个部署语义的分叉口。你跟 AI 说"部署方式由资源类型唯一决定"，指的就是这个函数。

实时同步黑科技

WebSocket Hub + 文件监听 = 资源变动，前端秒知

活字印刷 vs 飞鸽传书

传统 Web 页面要获取新数据，必须"主动问"服务器（刷新或轮询）。AiResourceManager 用 WebSocket 反其道行之——服务器"主动告诉"前端发生了什么。

📠

传统轮询

每 5 秒问一次"有新消息吗？" 99% 的请求是浪费

📡

WebSocket 推送

有变化时才通知。实时、省流量、低延迟

Hub：广播站

hub.go 是 WebSocket 连接的大脑。用 channel 实现 goroutine 间通信，保证并发安全。

CODE 

type Hub struct {
    clients   sync.Map
    register   chan *Client
    unregister chan *Client
    broadcast  chan []byte
    done       chan struct{}
}
PLAIN ENGLISH

定义 Hub 结构体

clients：线程安全的 Map，存储所有活跃连接

register channel：新客户端连入时发信号

unregister channel：客户端断开时发信号

broadcast channel：收到一条要推送给所有客户端的消息

done channel：服务关闭时，通知退出循环

为什么抑制机制很重要？

系统自己删除资源时会触发 fsnotify 的 REMOVE 事件。如果不加抑制，你会收到自己触发的"被删除"广播，误报成"用户在外部删了文件"。SuppressUUID() 就是解决这个先有鸡还是先有蛋的问题。

05

当两份配置撞车

Union-Find 算法 + 路径树语义判断 = 智能冲突检测

想象停车位

MCP 配置文件只有一个，但你想把多个 Config 资源部署进去。如果两份资源都试图占据同一个 key（比如在 mcpServers 下都叫 "database"），怎么决定谁让谁？

AiResourceManager 的答案：按 路径树语义，精确到嵌套位置判断冲突。

🧩
路径树语义

冲突 ≠ key 名字一样。冲突 = 同一个嵌套路径下 占据同一个 key 且至少一方不是 map。{a: {a1: 1}} vs {a: {a1: 2}} → 冲突。{a: {a1: {...}}} vs {a: {b1: {...}}} → a1 ≠ b1 分叉，不冲突。

冲突判断核心代码

deploy.go 第 974 行的 configsConflict 函数是判断冲突的根本逻辑：

CODE 

func configsConflict(a, b map[string]interface{}) bool {
    for k, av := range a {
        bv, ok := b[k]
        if !ok { continue }
        amap, aok := av.(map[string]interface{})
        bmap, bok := bv.(map[string]interface{})
        if aok &&amp;&amp; bok { continue }
        return true  // 冲突!
    }
    return false
}
PLAIN ENGLISH

定义函数：两个配置 map 是否冲突？

遍历配置 A 里的每个 key

看 B 是否有同名 key

B 没有 → 这个位置不冲突，跳过

两边都是 map → 可以深入合并，不算冲突

检查两边值的类型

任一方不是 map → 覆盖写入就是冲突！

遍历完没有发现冲突位

返回 false，这两份配置和平共处

函数结束

🎯
为什么数组不冲突？

{a: [1, 2]} vs {a: [3, 4]} — 数组走"拼接"语义，合并后变成 [1, 2, 3, 4]。所以两边都是 array 时不算冲突。这在代码里体现为第二个 if aArrOk && bArrOk { continue }。

Union-Find：把冲突资源分组

Union-Find 算法把有冲突的资源归到一个"冲突组"。组内按顺序最后一个胜出的标 applied，其余标 ignored。

1
两两比较所有待部署资源

如果有路径树语义冲突，union(i, j) 合并到同一组

2
检测多成员组

只有 >1 个资源的组才算冲突。单成员组直接 applied

3
检测与已部署内容的冲突

逐个待部署资源比对目标文件和已部署 Config，标记 existing

4
返回冲突报告

只有 ignored 或 existing 才算 hasRealConflict = true

Spot the Bug

找出下面简化版冲突判断的 bug：

1 func simpleConflict(a, b map[string]interface{}) bool {
2 for k, av := range a {
3 bv := b[k]; amap := av.(map[string]interface{})
4 bmap := bv.(map[string]interface{})
5 if amap && bmap { continue }
6 return true
7 }
8 return false
